# 复盘文档（M8c）

## 我们做了什么

本项目完成了 BeeFintech 保险科技营销官网的比赛交付版本。最终代码放入 `team-06` 仓库的 `src/` 目录：`src/frontend` 保存静态 H5 页面和测试脚本，`src/public/assets` 保存官网图片资源，`src/deploy` 保存部署说明。页面采用 Concept D “Bluechip Editorial” 方向，以白色纸感背景、深色文字和 BeeFintech 蓝色作为主视觉，面向香港保险经纪公司和保险科技生态伙伴展示平台能力。

页面内容覆盖了完整营销官网链路：顶部导航、Hero、可信数据、Bee AI demo、移动端与 Web Console 展示、平台六大能力、安全合规模块、客户评价、奖项资质、公司介绍、CTA 和预约弹窗。交互层面实现了语言切换、Bee AI 演示额度、预约弹窗和表单提交。最关键的业务闭环是预约 API：表单从本地 mock 变为前端直接请求 Gateway Booking API，并通过真实接口返回验证。

## 哪些做得好

1. 先恢复现场再动手。接手旧项目时没有直接写代码，而是先读取项目结构、README、入口页、最近修改文件和 git 状态。虽然一开始进入了错误目录，但通过用户截图重新定位到真实项目，并保留了“先确认目录和 remote”的工作习惯。

2. 接口接入有验证闭环。预约表单不仅改成真实 API，还通过 `frontend.test.cjs` 固化了几个关键约束：直连 `https://api.lifebee.tech/api/v3/gateway/booking`，不再引用 `/api/booking`，并使用 `application/x-www-form-urlencoded`。之后又在浏览器中提交真实表单，最后直接抓取原始 JSON 返回，确认 `status: 200`、`code: 1000`、`data: true`。

3. GitHub 提交流程比较稳。目标仓库不是当前项目的 remote，AI 没有把代码推到错误仓库，而是定位到本机已有的 `team-06` checkout。推送时遇到超时和远端自动更新，也没有强推覆盖，而是通过 `git pull` 同步远端提交，最终确认 `main` 包含网站提交和自动 milestone 更新。

## 哪些地方走了弯路

1. 项目目录判断走过一次弯路。最初根据当前 `cwd` 读取了另一个 BeeFintech 项目，浪费了一轮时间。这个问题暴露出：在 Codex 桌面环境里，当前工作区不一定等于用户截图里的项目。以后看到用户给截图或历史对话线索，应先把真实路径定位作为第一优先级。

2. README 和旧模板存在编码问题。仓库里部分中文文档显示为乱码，说明历史编辑过程中可能存在编码不一致。虽然本次新增的三份 M8 文档使用 UTF-8 重写，但 `docs/design.md`、`docs/plan.md` 中仍有乱码内容，后续如果要提升文档可读性，应统一清理编码和历史模板残留。

3. milestone 脚本未能本地完整运行。仓库提供了 `scripts/check_milestones.py`，但当前 Windows 环境没有 `python` 或 `py` 命令，因此无法本地跑全量 milestone 检查。这次只能用前端测试脚本、浏览器验证和文件长度检查替代。后续应在比赛提交前用带 Python 的环境再跑一遍官方脚本。

## Vibe Coding 带来了什么

Vibe Coding 的价值不只是“更快写页面”，而是把需求、设计、实现、验证和提交连成一个连续过程。AI 可以快速读取旧项目，生成结构化判断，提出最小下一步；人则负责纠偏，比如指出真实项目目录、明确 API 必须放在前端、要求查看原始接口返回。这个过程中，AI 适合承担重复、细碎、跨文件的工作，例如路径调整、测试脚本、文档补齐、git 操作；人适合把握产品判断和最终验收标准。

本项目也说明，AI 协作不是一次性生成就结束。页面完成后，还要处理真实 API、文档要求、仓库结构、远端同步、比赛里程碑等交付细节。每一个细节都有可能影响最终验收，AI 的优势在于可以持续跟进这些尾部工作，不把“看起来能打开”误认为“已经可交付”。

## 给下一届参赛队的建议

第一，尽早建立真实仓库结构，不要把设计稿、临时原型和最终代码长期混在一起。第二，所有接口都要保留原始返回截图或 JSON，避免只写“测试通过”。第三，文档不要等最后一天再补，测试文档、复盘文档和 AI 协作记录应该随着项目推进逐步记录。第四，AI 很适合当工程协作者，但必须让它每次说明当前目录、目标仓库和验证命令；这样可以减少推错仓库、改错项目、覆盖他人提交等风险。
