# 复盘文档（M8c）

## 1. 项目最终交付是什么

本项目最终交付的是 BeeFintech B2B 营销官网，而不是保险销售页、登录门户或后台系统。页面服务对象主要是香港保险经纪公司管理层、业务负责人、保险科技合作伙伴和比赛评委。网站的核心任务是建立品牌信任、解释平台能力、展示 Bee AI 演示体验，并把有兴趣的访客转化为预约演示线索。

最终页面采用单页长滚动结构，包含固定顶部导航、Hero 首屏、可信数字、Bee AI demo、产品与 Web Console 展示、平台能力、安全模块、客户模块、奖项资质、公司介绍、CTA 和预约弹窗。技术上使用静态 HTML / CSS / JavaScript，不依赖复杂前端框架，图片资源通过本地 `assets` 管理，预约接口采用前端直连 Gateway Booking API。

## 2. 需求和范围是怎么被修正的

项目中最重要的一次修正，是把“服务平台 / 门户 / 官网”的表述统一成“B2B 营销官网”。如果继续使用“门户”或“服务平台”，评委和后续开发者会自然期待登录、账号、子系统入口、业务数据查询等能力；但本期页面实际交付的是对外展示和预约转化。因此我们把平台能力放在官网中展示，而不是把官网本身写成一个业务系统。

第二次关键修正是合规风险。早期设计里有真实第三方品牌、保险产品名、具体金额和强背书式表达。经过风险检查后，页面不再新增未授权背书，不展示计划书生成流程，也不把 Bee AI demo 写成真实保险建议工具。这个取舍牺牲了一点“看起来很真”的冲击力，但更适合作为公开比赛作品。

第三次修正是预约表单。最初表单字段较多，包括公司、职位、关注模块、备注等，但当前 Gateway API 只支持 `name`、`phone`、`email`。如果继续保留多余字段，用户会误以为这些信息已经提交到后端。最终我们把表单压缩为 3 个字段，保证页面表单和真实接口能力一致。

## 3. 页面实现过程中做得好的地方

首先，页面调优不是只看代码，而是持续根据浏览器截图修正。首屏过大、导航不固定、顶部空白过高、AI demo 框太宽、Dashboard mock 在手机端只露出局部、客户模块语言没切换等问题，都是通过用户截图逐项暴露出来的。相比一次性生成页面，这种“看一屏、改一处、再看一屏”的方式更接近真实设计走查。

其次，Bee AI demo 的交互最终变得更适合演示。三个模式都提供默认输入，体验者不用自己想问题；点击发送后展示固定演示回答；不同模式之间保持独立状态；额度用完后自然引导预约。这个逻辑既保留了 AI 产品感，又避免了接入真实 AI 后端带来的不确定性。

第三，移动端最终没有被当成“顺手适配”。我们专门检查了手机宽度下的 Hero、Bee AI demo、Dashboard mock、客户卡片和预约弹窗。特别是 Dashboard mock，原本在移动端只露出局部，看起来像页面坏掉；后来改成整体缩放展示，让用户至少能理解这是一个网页控制台示意图。

## 4. API 接入的取舍

预约 API 是本项目最现实的业务闭环。我们最终选择前端直连 `https://api.lifebee.tech/api/v3/gateway/booking`，原因是本次比赛交付重点是官网前端和预约演示，不希望因为一个简单预约表单增加额外服务部署复杂度。

这个方案的好处是结构简单：静态官网可以直接发请求，字段清晰，调试成本低。代价是浏览器会受 CORS 和网关白名单影响，因此测试文档里必须明确写出：本地 `file://` 只能验证页面交互，最终部署域名仍需重新提交有效数据，确认网关返回 `code === 1000` 且 `data === true`。

这次也提醒我们，接口文档不能只写“已接入”。必须说明字段、请求格式、成功判断、失败风险和最终域名复测要求。否则页面上一个“提交成功”的提示，不等于真正完成了接口闭环。

## 5. 走过的弯路

第一个弯路是项目路径和上下文混淆。本地同时存在多个 BeeFintech 相关目录和比赛仓库，AI 一度把其他范围的内容带入当前新官网。这个问题的根源不是代码，而是工作现场没有先锁定。以后应该在任何修改前先确认三件事：当前项目路径、当前入口文件、当前交付范围。

第二个弯路是页面比例反复调整。用户希望整体更大，但首屏又必须在 100% 浏览器下完整可读。这个要求不是简单把页面 scale 到某个比例，而是要同时调整字号、间距、区块高度和右侧信息密度。最终经验是：营销官网的“高级感”不来自无限放大字体，而来自清楚的视觉层级和合适的留白。

第三个弯路是文档一开始太像模板。测试文档没有充分写移动端，复盘也没有体现真实决策过程。比赛文档不是凑字数，而是要让评委看见这个项目从需求、设计、实现、测试到复盘的完整路径。因此后续补文档时，重点改成了真实过程：定位修正、合规取舍、API 取舍、移动端测试和剩余风险。

## 6. Vibe Coding 的价值

这次项目体现的 Vibe Coding，不是让 AI 一次性替人完成所有东西，而是让 AI 参与连续协作：用户负责判断方向，AI 负责快速落地和整理证据。用户通过截图指出细节问题，AI 通过搜索文件、修改代码和补文档快速响应；用户再验收视觉和产品逻辑。这种循环让页面从“像一个官网”逐渐变成“可交付的比赛官网”。

AI 的优势是执行密度高，可以快速处理重复文案、语言字典、响应式样式、测试文档和复盘整理。人的优势是判断什么该保留、什么该删掉，例如保留旧版可信数字、去掉不该提的 proposal、压缩预约表单字段、决定不用额外中转服务。两者结合后，最终交付更接近真实产品工作流。

## 7. 后续建议

第一，最终发布前要在真实线上域名复测预约 API，记录一次有效请求的原始 JSON 返回。

第二，移动端应补充真机截图，尤其是 iPhone Safari、Android Chrome 和微信内置浏览器。

第三，继续清理所有文档里的历史乱码和范围混用表述，确保 `design.md`、`plan.md`、`test.md`、`ai-collab-log.md`、`retrospective.md` 讲的是同一个项目。

第四，如果后续要从比赛 demo 走向正式官网，需要重新审查第三方品牌、产品名、数字声明、客户证言和隐私说明，确保所有公开表达都有授权或可审计依据。
