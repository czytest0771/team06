# Brainstorm（M2）

> 本文档整理 BeeFintech 营销官网在需求定义阶段的关键 Q&A。内容来自 `docs/design.md` 中已经确定的目标、页面架构、核心功能和验收标准，用于说明为什么最终选择当前官网方案。

## Q1: 这个官网最重要的访问者是谁？

A: 最重要的访问者不是普通 C 端保险消费者，而是保险经纪公司老板、业务负责人、运营负责人、保险机构渠道团队、IT 负责人和合规负责人。页面需要让这些人快速理解 BeeFintech 是面向保险经纪业务的保险科技平台，而不是单一保险产品销售页。因此首页首屏必须强调平台、经纪业务、可信数据和企业级能力。

## Q2: 用户进入页面后 30 秒内必须理解什么？

A: 用户需要在 30 秒内判断三件事：BeeFintech 服务谁、解决什么问题、是否值得预约演示。具体来说，页面要传达“面向香港保险经纪业务”“覆盖销售、签单、服务、运营与佣金管理”“具备 Bee AI、移动端和 Web Console 的完整产品能力”。这也是 Hero、Trust stats、Platform 能力和产品展示被放在前半段的原因。

## Q3: 为什么选择单页长滚动官网结构？

A: 比赛 demo 的目标是快速展示完整品牌和产品叙事，单页长滚动比多页面更适合评委和潜在客户快速浏览。页面从品牌认知开始，经过 Bee AI 体验、产品能力、安全与客户信任，再到公司信息和预约转化，形成一条完整路径。顶部导航提供锚点跳转，既保留单页效率，也方便用户直接查看 Platform、AI Broker、Security、Customers 和 Company。

## Q4: 为什么一定要有 Bee AI demo？

A: Bee AI demo 是页面的差异化体验。如果只有静态文案，用户很难感受到保险科技平台的智能服务能力。公开官网中的 Bee AI 不接入完整产品库，而是用 3 次 credit 限制和固定演示回答，让用户看到“产品对比、方案可视化、合规问答”的方向。额度用完后引导预约，既控制 demo 风险，也把兴趣转化为销售线索。

## Q5: 预约表单承担什么角色？

A: 预约表单是官网的主要转化动作。所有 `Book a demo` 入口都应复用同一套弹窗或表单逻辑，避免页面上出现多套不一致表单。最终字段聚焦为姓名、电话和邮箱，并通过 Gateway Booking API 提交。成功后显示“已收到预约，我们会在一个工作日内联系您”的反馈，让用户明确提交已经被接收。

## Q6: 为什么页面要强调安全、客户和奖项资质？

A: BeeFintech 的目标客户是保险经纪和机构团队，决策时会关注数据安全、合规、可靠性和外部认可。安全模块用于回答数据驻留、加密、审计、PDPO 等问题；客户评价用于表达业务价值；奖项资质用于建立第三方背书。它们共同解决“这个平台是否可信”的问题，而不是单纯装饰页面。

## Q7: 为什么技术实现选择静态 HTML/CSS/JavaScript？

A: 比赛交付要求是快速、可访问、可提交。当前官网不需要复杂后台渲染，也不需要完整前端框架。静态 H5 能降低部署成本，便于直接放入 `src/frontend/index.html`，并通过 `src/public/assets` 管理图片资源。交互部分用原生 JavaScript 实现语言切换、Bee AI demo、预约弹窗和 API 提交，足以满足 MVP 验收。

## Q8: 移动端适配最关键的风险是什么？

A: 移动端最关键的风险是横向溢出和表单不可用。页面有大标题、三列卡片、手机 mockup、浏览器 mockup 和奖项图片，如果不处理断点，很容易在手机上出现横向滚动。移动端需要保证导航可访问、Bee AI demo 宽度占满容器、预约弹窗能完整填写和关闭、Platform 和 Awards 卡片能从多列变成单列或两列。

## Q9: 这个 demo 与生产版本的边界在哪里？

A: 当前页面是 Vibe Coding 比赛 demo，不等于生产官网。设计文档中明确说明部分品牌 logo、产品名、价格、客户评价和 stats 需要法律或业务审核。生产化时应替换真实授权素材、统一内容来源、补充正式部署流程，并根据业务需要决定 Bee AI 是否接入真实服务。比赛版本重点证明页面架构、交互闭环和 API 接入能力。
