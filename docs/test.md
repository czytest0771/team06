# 测试文档（M8a）

## 1. 测试对象

本次测试对象是 BeeFintech B2B 营销官网当前新版本。网站为单页 H5 官网，入口文件为 `index.html`，核心模块包括：固定顶部导航、Hero 首屏、可信数字、使用场景条、Bee AI demo、移动端与 Web Console 展示、平台能力、安全模块、客户模块、奖项资质、公司介绍、CTA 和预约演示弹窗。

本轮测试只围绕当前新官网，不包含其他归档页面或非本期范围功能。预约表单当前采用前端直连 Gateway Booking API 的方案，字段只保留 `name`、`phone`、`email`。

## 5. Bee AI demo 测试

Bee AI demo 是官网内的轻量演示组件，不调用真实 AI 服务。当前保留三个模式：

1. Compare products / 比较产品
2. Visualize a scenario / 可视化方案
3. Ask Bee anything / 向 Bee 提问

测试步骤：

1. 打开页面并跳转到 Bee AI demo 区域。
2. 检查初始额度为 `3 / 3`。
3. 在“比较产品”模式下不手动输入，直接点击发送。
4. 页面使用默认输入内容提交，并展示对应的固定回答。
5. 切换到“可视化方案”，输入框恢复该模式默认内容，回答区回到未回答状态。
6. 再切换到“向 Bee 提问”，同样恢复该模式默认内容和未回答状态。
7. 连续提交 3 次后，额度变成 `0 / 3`。
8. 额度用完后再次点击发送，打开预约演示弹窗。

测试结果：通过。该设计降低了体验者不知道输入什么的问题，也保证三个模式互不干扰。

## 6. 预约表单与 Gateway API 测试

预约弹窗字段为：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `name` | 是 | 预约人姓名 |
| `phone` | 是 | 联系手机号 |
| `email` | 是 | 联系邮箱 |

当前前端直接请求：

```text
POST https://api.lifebee.tech/api/v3/gateway/booking
Content-Type: application/x-www-form-urlencoded
```

请求体示例：

```text
name=Test User&phone=13333333333&email=test@example.com
```

成功判断：响应 JSON 中 `code === 1000` 且 `data === true`。页面成功后应显示“已收到您的预约，我们会在一个工作日内联系您。”；失败时显示错误提示，不应误报成功。

测试注意事项：

- 过短姓名、无效邮箱或重复手机号可能被网关拒绝。
- 因为是前端直连，最终部署域名仍需复测 CORS。
- 本地 `file://` 可以测试页面交互，但不能完全代表线上网络环境。

### 7.4 移动端已知限制

当前移动端主要基于浏览器模拟视口完成测试。模拟结果已覆盖布局、字号、横向滚动和主要交互，但真实手机仍可能出现地址栏高度、系统字体渲染、微信内置浏览器兼容性差异。因此提交前建议补充至少两类真机截图：iPhone Safari / Chrome 与 Android Chrome / 微信内置浏览器。

## 8. 已知风险与后续验证

1. 前端直连 Gateway API 仍需在最终部署域名复测 CORS。
2. Gateway 对表单数据可能有格式和重复提交限制，测试时应使用有效且唯一的数据。
3. 移动端已经完成模拟测试，但真机测试仍建议补充。
4. `design.md`、`plan.md`、`test.md`、`ai-collab-log.md`、`retrospective.md` 应在最终提交前统一检查编码和范围表述。
