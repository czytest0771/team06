# AI 协作记录文档（M8b）

## 1. 协作目标

本项目使用 AI 协助完成 BeeFintech B2B 营销官网的比赛交付。网站目标是面向香港保险经纪公司、保险科技合作伙伴和比赛评委，展示 BeeFintech 的品牌可信度、平台能力、Bee AI demo、安全说明、客户信任和预约演示入口。

本轮 AI 协作不是单纯让 AI 一次生成页面，而是围绕“需求澄清 → 设计判断 → 前端实现 → 体验调优 → 接口接入 → 移动端适配 → 文档补齐”的完整流程展开。用户负责产品方向、设计取舍和最终验收，AI 负责快速查找文件、梳理逻辑、修改实现、生成文档和补充测试说明。

## 2. 使用的 AI 工具与方式

主要使用 Codex 作为工程协作工具，配合 superpowers 系列方法完成需求和计划文档：

- `superpowers:brainstorming`：用于早期整理官网定位、目标用户、核心页面模块和交互范围。
- `superpowers:writing-plans`：用于把需求拆成阶段计划，形成 `plan.md` 的执行结构。
- `grill-me / grill-with-docs`：用于对文档进行追问和风险检查，发现“网站 / 门户 / 服务平台”混用、用户类型混杂、合规风险、预约字段不匹配等问题。
- Codex 工程协作：用于读取项目文件、定位页面入口、修改 `index.html`、检查语言切换、补测试文档和复盘文档。

AI 的角色更像“执行型产品工程助理”：它不直接决定产品方向，而是在用户明确判断后，把判断落实到页面和文档里。

## 3. 阶段一：需求和定位澄清

早期文档里曾出现“保险科技 Web 服务平台”“官网升级”“服务门户”等不同说法。AI 在 grill 过程中指出，这些词对应的产品边界不同：官网偏展示和转化，门户意味着登录和子系统入口，服务平台则意味着访客可以直接消费业务能力。

用户最终确认本期更接近 B2B 营销官网。于是文档和页面表达都围绕“展示 BeeFintech 的保险科技能力，并引导预约演示”展开，不把当前网站写成登录门户或完整业务系统。这一轮协作避免了范围被写大，也让后续测试标准更明确：评委要看到的是可访问、可理解、可演示、可预约的官网，而不是一个未完成的业务平台。

## 4. 阶段二：合规和内容风险检查

AI 对设计稿和页面内容做过一次风险审查。重点检查了真实第三方品牌、保险产品名、具体金额、真实姓名、客户证言、奖项背书和“Approved by”这类措辞。用户也特别强调网页不要再提 proposal 相关内容，避免让 Bee AI demo 看起来像真实计划书生成或保险建议。

最终协作结果是：页面保留官网需要的可信表达和演示感，但减少不必要的合规风险。Bee AI demo 被定义为公开演示组件，不是事实调查工具，也不是正式保险建议。预约表单也从复杂字段收敛为真实 Gateway API 支持的 `name`、`phone`、`email` 三个字段，避免用户填写了无法提交的内容。

## 5. 阶段三：设计和前端体验迭代

前端页面并不是一次成型，而是通过用户截图持续调优。AI 每次根据截图定位对应模块，再做小范围修改。比较关键的迭代包括：

- 顶部导航改为固定，方便用户下滑后随时切换模块。
- 导航增加 Home，并调整顺序为 Home、AI 经纪助手、平台、安全、客户、公司，使导航顺序和页面内容一致。
- 首屏字号、整体缩放和留白多次调整，目标是在浏览器 100% 缩放下既有冲击力，又能看到完整首屏信息。
- Hero 可信数字恢复为旧版内容，保持“95%+ / 8 yrs / 60%+ / 20,000+”这组强记忆点。
- 客户模块恢复旧版结构，并修复语言切换未覆盖的问题。
- Dashboard mock 调整缩放和位置，避免右侧内容在 100% 浏览器下完全看不见。
- 移动端将网页控制台示意图整体缩放展示，避免只露出局部导致用户误以为页面坏掉。

这一阶段中，用户提供了大量视觉反馈，AI 负责把反馈转成 CSS、HTML 和文案层面的具体修改。

## 6. 阶段四：Bee AI demo 交互协作

Bee AI demo 是本页最重要的交互模块。用户提出了几个关键体验要求：

1. 只有提交问题后才展示回答，未提交时不应默认出现回答区。
2. 三个模式分别是“比较产品”“可视化方案”“向 Bee 提问”，不再保留 proposal 相关按钮。
3. 输入框要有默认问题，用户可以不改内容直接点击发送。
4. 回答语言必须跟随网站语言切换。
5. 每个模式应有独立状态：在比较产品里提交后，切到可视化方案或向 Bee 提问，应回到该模式的默认输入和未回答状态。
6. 额度从 `3 / 3` 开始，每次提交扣减一次，用完后再提交打开预约弹窗。

AI 根据这些要求调整了状态逻辑和语言字典，让 demo 更适合比赛现场展示。这个模块最终体现了“演示优先”的设计思想：它不追求真实 AI 后端能力，而是让访客快速理解 Bee AI 可以做什么，并自然进入预约转化。

## 7. 阶段五：预约 API 协作

预约 API 是本项目的真实业务闭环。用户明确当前新官网采用前端直连 Gateway API，不增加额外中转服务。AI 根据接口文档确认请求方式：

```text
POST https://api.lifebee.tech/api/v3/gateway/booking
Content-Type: application/x-www-form-urlencoded
```

字段为：

```text
name
phone
email
```

AI 协助把预约弹窗字段压缩为这三个字段，并把成功判断写成围绕 Gateway 返回值：`code === 1000 && data === true`。同时在测试说明中记录前端直连的风险：最终部署域名仍需复测 CORS，测试数据也要避免过短、无效或重复。

这个阶段最大的收获是：接口接入不能只看页面提示“提交成功”，还要看真实 API 返回。用户多次追问错误原因，推动 AI 区分网络错误、CORS 风险、业务失败和测试数据不合法，最终文档也把这些风险写清楚。

## 8. 阶段六：移动端适配协作

移动端适配一开始不是文档重点，但在用户检查页面后发现：Dashboard mock 在手机端只露出局部，看起来不完整；AI demo 卡片在 100% 宽度下也需要进一步收紧。之后 AI 按用户要求调整移动端显示方式：

- Hero 从桌面两栏改为移动端纵向阅读。
- Bee AI demo 卡片收窄，保证输入框和发送按钮在手机屏幕内可见。
- Dashboard mock 在移动端整体缩放，而不是裁切局部。
- 客户卡片、平台能力卡片、预约弹窗改成更适合小屏的单列结构。

这一轮协作后来也反映到 `test.md` 中，补充了 360px、390px、430px 和 768px 的移动端测试项。用户指出测试文档没有充分写移动端后，AI 又补充了移动端页面结构检查、移动端交互检查和真机复测建议。

## 9. 阶段七：比赛文档补齐

最后阶段，AI 协助重写比赛文档，重点是 `test.md`、`ai-collab-log.md`、`retrospective.md`。用户指出原文档不够满意：测试文档移动端写得太少，复盘文件像模板，AI 协作记录也需要更完整。

于是文档被改成更贴近真实项目过程：

- `test.md`：补充桌面、多语言、Bee AI demo、预约 API、移动端适配和已知风险。
- `retrospective.md`：从项目定位、合规取舍、页面调优、API 方案、移动端适配、Vibe Coding 价值几个角度复盘。
- `ai-collab-log.md`：按协作阶段记录 AI 和用户如何共同推进项目，而不是只列工具名称。

## 10. 协作复盘

这次 AI 协作最有效的地方，是让 AI 承担高频、细碎、跨文件的执行工作：查路径、搜文案、改 CSS、补语言字典、整理文档、验证接口。用户则持续做关键判断：到底做营销官网还是门户，要不要提 proposal，哪些旧版内容要保留，预约接口用不用中转，移动端看起来是否合理。

这次协作也暴露了一个问题：如果项目路径和范围没有先锁定，AI 很容易把其他上下文带进来。因此后续使用 AI 时，应先让 AI 明确三件事：当前项目路径、当前入口文件、当前交付范围。只有这个基础稳定，AI 的执行效率才不会变成误改风险。

总体来看，AI 在本项目中不是替代产品经理、设计师或工程师，而是把他们的判断快速落地。用户越能清楚指出“这里不对”“这个保留旧版”“这个不要提”，AI 越能把页面和文档推向可交付状态。
